## 技术决策

- **协议级 ping 而非 JSON ping/pong**：Starlette 原生支持 `ws.ping()`，零开销，不干扰应用层消息流
- **Session 清理不持有 WS 引用**：避免循环引用；连接断开时 finally 已清理
- **运动检测在 handler 层过滤**：在存储 latest_frame 之前过滤，不影响下游 consumer
- **pytest + Starlette TestClient**：httpx 0.28 的 WS 支持不稳定，用 Starlette 内置 TestClient 更可靠
- **VAD 独立音频流**：用 `MediaStreamTrack.clone()` 给 VAD 创建隔离的 `vadStream`，避免 vad-web 的 `destroy()`/`pause()` 停止视频 track 导致摄像头黑屏

## 遇到的问题

### Camera/Mic toggle 摄像头黑屏（多次迭代修复）
- **问题 1**：`track.stop()` 后未 `removeTrack()`，旧死轨道残留导致 `new MediaStream()` 转移轨道时画面丢失
- **问题 2**：`new MediaStream()` 隐式转移 track 使 `streamRef` 指向空流
- **问题 3**：`srcObject` 设置同一引用 Chrome 不刷新 => 用 `srcObject=null → srcObject=stream` 强制重绑
- **根因**：vad-web 的 `destroy()` / `pause()` 会停止 stream 上**全部** track（含 video track）。即使只开关麦克风，视频 track 也被杀掉
- **最终修复**：用 `audioTrack.clone()` 创建独立的 `vadStream`（纯音频流）给 VAD 使用，主 `stream` 专供 `<video>` 和 frame capture。两个流完全隔离

### Ollama 推理延迟 ~17s
- transcript → Ollama 请求仅 3ms（代码无瓶颈）
- 延迟全在 Ollama 推理内部，推测为模型冷加载（长时间闲置后模型从内存卸载）
