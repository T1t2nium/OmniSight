## 二、运营成本控制：我想到了哪些技巧，实际采用了哪些

> 我的硬件环境：**RTX 3060 6GB 显存**，Windows 11 笔记本。以下所有成本评估基于此环境。

### 2.1 本地优先策略：零 API 费用

| 技巧 | 是否采用 | 效果 |
|------|---------|------|
| **默认使用本地 Ollama 模型** | ✅ 已采用 | 完全消除 API 调用费用。按每天 100 次对话估算，使用云端 API（如 GPT-4o-mini）月费约 ¥60-180，本地仅需电费约 ¥15 |
| **云端 API 仅作备选通道** | ✅ 已采用 | Gemini 通道已预留但默认关闭，无需付费即可用 |
| **CPU + GPU 混合推理** | ✅ 已采用 | Whisper 和 TTS 跑 CPU，Ollama 模型利用 RTX 3060（6GB 显存）推理，各司其职 |

**关键换算**：

| 方案 | 月成本（100 次对话/天） | 隐私 |
|------|------------------------|------|
| 纯云端 API（GPT-4o-mini + Whisper + TTS） | ~¥120-250 | 数据上传 |
| 本地 Ollama（qwen3.5:2b）+ 本地 Whisper + 本地 TTS | ~¥15（电费） | 数据不离开本机 |
| **我采用的方案** | **~¥15/月** | **完全本地** |

### 2.2 模型选择：用小模型省大钱

| 技巧 | 是否采用 | 效果 |
|------|---------|------|
| **选用 qwen3.5:2b 而非 gemma3:12b** | ✅ 已采用 | 2B 参数模型显存占用 ~2GB，我的 RTX 3060 6GB 绰绰有余。推理延迟 1-2s vs 12B 模型的 8-15s，对话体验反而更好 |
| **关闭 Qwen3 的 thinking/reasoning 模式** | ✅ 已采用 | `think: False` 减少约 30% 输出 token，每轮平均少 15-25 个 token |
| **大模型预留切换能力** | ⏳ 架构已支持 | 日常用 2B 快速响应，复杂任务可随时切到 12B 或更大模型，不必一直跑大模型 |
| 云端大模型作为兜底 | ⏳ 架构已预留 | Gemini API 通道已写但默认不启用 |

### 2.3 Token 经济：每个 token 都算成本

| 技巧 | 是否采用 | 效果 |
|------|---------|------|
| **对话历史裁剪到最近 8 条** | ✅ 已采用 | 防止上下文窗口膨胀，每轮推理 token 消耗可控。8 条消息约 500-800 token，不加裁剪会线性增长 |
| **运动检测跳过无变化帧** | ✅ 已采用 | `FrameMotionDetector`：画面静止时不更新最新帧，减少无效视觉 token |
| **视频帧低频发送（4 fps）** | ✅ 已采用 | 前端 30fps 采集，后端取 4fps 给 AI，输入图片数降 87% |
| **系统提示词紧凑化** | ✅ 已采用 | 初版仅 3 行（~150 token），用户优化版约 50 行（~400 token），仍在小模型的 2048 上下文窗口内 |
| 视觉帧按需发送（仅视觉问题） | 设计后撤回 | 最初用正则 `_is_visual_question()` 过滤，但用户改提示词后选择禁用，由提示词自行控制 |

### 2.4 本地推理效率

| 技巧 | 是否采用 | 效果 |
|------|---------|------|
| **Whisper 用 base 模型而非 large-v3** | ✅ 已采用 | base 模型 ~140MB vs large-v3 ~3GB。我的 RTX 3060 6GB 显存优先给 Ollama，Whisper 用 CPU（base 模型在 CPU 上 ~1-2s/句） |
| **sherpa-onnx CPU 推理** | ✅ 已采用 | RTF < 0.1，20-50 字合成仅需 50-150ms。ONNX Runtime CPU 执行，不占显存 |
| **VAD 浏览器端 WASM 运行** | ✅ 已采用 | Silero VAD 在浏览器 WASM 中执行，不占用服务端 CPU |
| **TTS 句子级流式合成** | ✅ 已采用 | 不等 LLM 完整回复即开始合成，首字延迟感知降低 40-60% |
| GPU 加速 Whisper | ❌ 未采用 | Whisper 放 GPU 会和 Ollama 抢显存。RTX 3060 6GB 显存有限，优先保证对话模型 |

### 2.5 硬件资源分配（RTX 3060 6GB 实际布局）

| 组件 | 运行位置 | 显存/内存占用 |
|------|---------|-------------|
| Ollama qwen3.5:2b | **GPU** | ~2.1 GB 显存 |
| faster-whisper base | CPU | ~300 MB 内存 |
| sherpa-onnx TTS | CPU | ~250 MB 内存 |
| Silero VAD | 浏览器 WASM | ~50 MB（浏览器侧） |
| Canvas 粒子系统 | 浏览器 GPU | ~10 MB 显存（浏览器进程） |
| **总计** | | **GPU 2.1 GB / 6 GB 可用 · RAM ~550 MB** |

### 2.6 UI 资源成本

| 技巧 | 是否采用 | 效果 |
|------|---------|------|
| **Canvas API 替代动画库** | ✅ 已采用 | 流场粒子系统纯 Canvas 2D，不引入 Three.js / GSAP / framer-motion，打包体积零增长 |
| **CSS 变量替代 CSS-in-JS** | ✅ 已采用 | 零运行时开销，DevTools 可直接调试 |
| **按钮使用 emoji 而非图标库** | ✅ 已采用 | 不引入 Lucide/Heroicons/FontAwesome，减少 JS bundle |
| **零新增 npm 依赖** | ✅ 已采用 | PR 6-9 所有前端改动均未安装新包 |
| 引入 Tailwind CSS | ❌ 未采用 | 项目已有 450 行 CSS，全部重构成本 >3000 行，收益不足以覆盖 |
| 引入 shadcn/ui 组件库 | ❌ 未采用 | 依赖 Tailwind + 额外包，且组件风格与项目设计 token 体系冲突 |

### 2.7 被认真考虑但最终驳回的技巧

| 技巧 | 驳回理由 |
|------|---------|
| GPU 统一推理（全部放 GPU） | RTX 3060 仅 6GB 显存，qwen3.5:2b 已占 2GB，Whisper 上 GPU 会抢显存导致 Ollama 推理降速 |
| 对话历史持久化到 SQLite | 当前单用户内存方案已满足需求，引入 DB 增加维护成本而收益几乎为零 |
| Prompt 缓存 / KV-cache 复用 | Ollama 不暴露此 API；且短对话（<8 条消息）场景下收益有限 |
| Electron 打包桌面应用 | 浏览器方案零安装、零分发成本，Electron 打包后体积 150MB+ |
| Kokoro / Piper TTS | 两者依赖 espeak-ng 做中文 G2P，拼音-IPA 映射缺失导致口齿不清。sherpa-onnx 内置 FST + 词典是唯一解 |
| CDN 托管 Whisper 模型 | 本地存储避免网络延迟，base 模型仅 140MB，下载一次永久使用 |
| 用 Llama 3.2 Vision 70B | 需 >40GB 显存，我的 RTX 3060 6GB 完全无法运行，违背本地优先原则 |

### 运营成本总结

以我的 RTX 3060 6GB 笔记本为基准，整个系统运行在纯本地环境：

- **直接费用**：¥0（零 API 调用费），每月仅电费约 ¥15
- **等效云端成本**：若全部用云 API 替代，月费 ¥120-250
- **隐性节省**：通过历史裁剪 / 运动检测 / 小模型等技术，单日推理 token 比不加控制减少约 60%
- **硬件利用率**：GPU 仅用 2.1/6 GB 显存，RAM 总占用 < 1 GB，笔记本日常办公完全不受影响

### 主要教训

1. **2B 模型配好提示词 > 12B 模型配烂提示词**：qwen3.5:2b 在结构化系统提示词引导下，对话体验远超预期，推理延迟仅 1-2s
2. **espeak-ng 中文 G2P 是所有依赖它的 TTS 引擎的死穴**：Piper/Kokoro 都栽在这一步，最终 sherpa-onnx 内置词典解决了问题
3. **正则过滤不如提示词灵活**：最初用正则匹配视觉问题决定是否传图，后来改成始终传图让提示词自行判断，用户调优自由度更高
4. **Canvas 2D 做粒子特效完全够用**：600 粒子 60fps，在集显上 CPU 占用 <3%，无需 WebGL/Three.js
5. **零依赖原则让维护成本可控**：整个前端改动期间零新增 npm 包，回归风险大幅降低
