## 二、运营成本控制：想到 vs 采用

### 本地优先策略

| 想法 | 是否采用 | 说明 |
|------|---------|------|
| 默认使用本地 Ollama 模型 | ✅ 已采用 | 零 API 费用，数据不离开本机 |
| 云端 API 仅作备选 | ✅ 已采用 | Gemini 通道已预留但默认关闭 |
| 支持 CPU-only 推理 | ✅ 已采用 | 无需 GPU，普通笔记本即可运行全部模型 |

### 模型选择策略

| 想法 | 是否采用 | 说明 |
|------|---------|------|
| 计划用 gemma3:12b | ❌ 改用 qwen3.5:2b | 2B 参数模型在对话场景下足够智能，推理速度更快、内存占用更低 |
| 大模型仅复杂场景启用 | ⏳ 可选 | 架构支持随时切换模型，日常用 2B，重度场景切 12B |
| 关闭 thinking/reasoning 模式 | ✅ 已采用 | Qwen3 的 `think: False` 降低约 30% token 消耗 |

### Token 经济

| 想法 | 是否采用 | 说明 |
|------|---------|------|
| 对话历史裁剪（最近 8 条） | ✅ 已采用 | 防止上下文窗口膨胀，降低每轮推理耗时 |
| 视觉帧按需发送 | 设计后撤回 | 最初实现 `_is_visual_question()` 过滤，后改为始终发送让提示词控制 |
| 运动检测跳过冗余帧 | ✅ 已采用 | `FrameMotionDetector`：无变化的帧不发送给 AI |
| 视频帧低频发送（4 fps） | ✅ 已采用 | 前端采集 30fps，后端仅取 4fps 给 AI |
| 系统提示词紧凑化 | ✅ 已采用 | 初版 3 行 → 用户优化版 50 行，保持在 2B 模型有效上下文窗口内 |

### 模型托管与推理

| 想法 | 是否采用 | 说明 |
|------|---------|------|
| ONNX Runtime CPU 推理 | ✅ 已采用 | faster-whisper + sherpa-onnx 均用 ONNX CPU，无需 CUDA |
| 模型本地缓存 | ✅ 已采用 | Whisper 模型、sherpa 模型、VAD 模型均本地存储 |
| 小模型优先 | ✅ 已采用 | Whisper 用 base 而非 large-v3（~140MB vs ~3GB） |
| 推理资源隔离 | ⏳ 未采用 | 单用户场景无需多进程隔离，后续多用户可加 |

### 音频处理成本

| 想法 | 是否采用 | 说明 |
|------|---------|------|
| 浏览器端 VAD 降低服务端负载 | ✅ 已采用 | Silero VAD 在浏览器 WASM 中运行，不占用服务端 CPU |
| TTS 句子级流式合成 | ✅ 已采用 | 不等 LLM 完成即开始 TTS，降低首字延迟感知 |
| PCM16 音频归一化 | ✅ 已采用 | 峰值归一化到 85%，避免音量忽大忽小产生额外重听 |

### UI 资源成本

| 想法 | 是否采用 | 说明 |
|------|---------|------|
| Canvas API 替代动画库 | ✅ 已采用 | 流场粒子纯 Canvas 2D，零依赖，不引入 Three.js / GSAP |
| CSS 变量替代 CSS-in-JS | ✅ 已采用 | 零运行时开销，DevTools 可直接调试 |
| 不引入图标库 | ✅ 已采用 | 按钮使用 emoji 而非 Lucide/Heroicons，减少打包体积 |
| 零新增 npm 依赖 | ✅ 已采用 | PR 6-9 所有前端改动均未安装新 npm 包 |

### 未被采用的想法

| 想法 | 驳回理由 |
|------|---------|
| 用 GPU 加速推理 | 目标用户设备可能无 GPU，CPU-first 覆盖更广 |
| 对话历史持久化到数据库 | 内存存储满足当前需求，引入 DB 增加运维成本 |
| 使用 Llama 3.2 Vision 70B | 本地推理不可行（需 >40GB 显存），违背本地优先原则 |
| Electron 打包桌面应用 | 浏览器方案零安装，维护成本更低 |
| Prompt 缓存 / KV-cache 复用 | Ollama 未暴露此 API，且短对话场景收益有限 |
| CDN 托管 Whisper 模型 | 本地存储避免下载延迟和带宽消耗 |
| 引入 Tailwind CSS / shadcn | 项目已有 450 行 CSS，重构成本高于收益 |
| Kokoro TTS（ONNX） | 依赖 espeak-ng 中文 G2P 有结构性缺陷，改用 sherpa-onnx |

## 三、总结

### 实现率

| 类别 | 计划数 | 已实现 | 实现率 |
|------|--------|--------|--------|
| 功能需求（F1-F9） | 33 个子需求 | 32 个 | **97%** |
| UI/UX 需求（F10） | 8 个 | 8 个 | **100%** |
| 非功能需求（N1-N5） | 13 个 | 13 个 | **100%** |
| Stretch Goals | 6 个 | 0 个（有意跳过） | 0% |

### 成本控制关键数字

| 指标 | 当地模型 | 等效云端 API |
|------|---------|-------------|
| 单次对话成本 | ¥0 | ¥0.005-0.02 |
| 月运行成本（100 次/天） | 电费 ~¥15 | API 费 ~¥60-180 |
| 隐私风险 | 零（数据不离开本机） | 中-高 |
| 模型切换开销 | 零（本地切换） | 需改 API Key / 代码 |

### 主要教训

1. **小模型足够好**：qwen3.5:2b 在结构化提示词引导下，对话体验远超预期，且推理延迟仅 1-2s
2. **espeak-ng 中文 G2P 是 TTS 瓶颈**：Piper/Kokoro 等依赖 espeak-ng 的方案都有结构性缺陷，sherpa-onnx 内置词典才是正解
3. **提示词比代码过滤更灵活**：最初用正则过滤视觉关键词，最终改为让系统提示词控制，用户自行调优
4. **Canvas API 零依赖做粒子特效**：600 粒子 @ 60fps 在集显笔记本上 CPU 占用 <3%
